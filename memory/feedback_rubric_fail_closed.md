When a rubric factor concerns a security-sensitive invariant, do not collapse "surfaces a warning" and "fails closed" into one factor. They are different enforcement layers (prompt text vs gate-state transition) and the executor will optimize the easier one.

**Why:** On PR #155 (issue #148, 2026-04-12), rubric factor 3 was worded "`loadRubricFromRunDir` surfaces missing rubric as visible warning, not silent null." The executor implemented the visible warning. Relay-review LGTM'd round 1. Post-merge codex adversarial review surfaced that the warning was only rendered in the reviewer prompt text — the review-runner state transition still advanced to `ready_to_merge` on a PASS verdict regardless of rubric state. "Visible" got optimized for; fail-closed gating didn't. Same compliance-theater failure mode as #138 round 5, one layer up.

**How to apply:** For security-sensitive rubrics, split factors by enforcement layer — e.g., "warning rendered in reviewer prompt" AND separately "state transition refuses to advance when invariant unsatisfied." Also: for any invariant that has a "trust root" (a value derived from input data), add a factor that validates the trust root itself — path-containment factors need a companion factor validating the base path. The #148 rubric validated `rubric_path` containment but not `run_id`, leaving the whole scheme forgeable via a crafted `run_id` (filed as #156). **Sibling-field extension (2026-04-12, from codex challenge on merged #156/PR #159):** when adding the trust-root companion factor, also enumerate every SIBLING field in the same logical scope that feeds the same downstream consumers. #156 pinned `run_id` but `paths.repo_root` / `paths.worktree` are adjacent trust roots that steer the same dispatch/review/merge/cleanup operations — filed as #160. The meta-rule: for trust root X, list all fields at the same level of the manifest schema that feed the same filesystem/GitHub operations, and each must be either validated in this PR or explicitly deferred with a follow-up issue.

**State-machine axis enumeration (2026-04-12, from codex challenge on merged #165/PR #167):** the sibling-enumeration meta-rule from #156/#160 applies to the state-machine axis too, not just trust-root fields. When fixing a state-machine-gated invariant, enumerate EVERY state at the relevant enforcement point, not just the state cited in the original bug report. #149 excluded merged/closed from branch-fallback. #165 added escalated exclusion for the specific case codex surfaced. Codex's post-merge challenge of #165 showed `dispatched`, `review_pending`, `changes_requested`, `ready_to_merge` all have the same stale-inheritance vector when `pr_number: null`. Filed as #168. The "specific instance" fix pattern (only the state cited in the bug) is a compliance-theater tell: when the rubric says "exclude state X from path Y", check the OTHER states at path Y explicitly; most of the time, the right fix treats the state axis as a whitelist (only `dispatched` is legitimate for null pr_number), not a blacklist.

**Recovery-path end-to-end test extension (2026-04-12, from codex challenge on merged #157/PR #162):** when a rubric factor covers "operator recovery" or "actionable next steps," it MUST require an end-to-end executable test of the full recovery flow, not just verification that the target state is non-terminal or that recovery text is printed. #157 filled in the "fail-closed gate" half of #155's compliance-theater pattern (warning vs gate), but factor 5 [operator communication] required only that recovery text name the path. The shipped fix documented `dispatch --run-id` as recovery, but `dispatch.js:464` requires `state=changes_requested` and the fail-closed gate leaves state at `review_pending` — recovery is unreachable. Filed as #163. Same compliance-theater pattern recurred one layer deeper: "visible" (closed in #148) → "fail-closed" (closed in #157) → "state non-terminal" (closed in #157 round 4) → *actually recoverable via real operator commands* (open in #163). Rubric rule for any "operator recovery" factor: require a regression test that exercises the full documented flow end-to-end, including the dispatcher/tool invocations, not just the state-machine transition.

**Selector-composition axis enumeration (2026-04-13, from codex challenge on merged #168/PR #169):** the state-machine-axis whitelist meta-rule (added after #167) applies to EVERY resolver selector that feeds the same `matches` variable, not just the one surfaced in the bug. #149 closed terminal exclusion in `filterByBranch`. #165 / #168 closed state-machine whitelist in `filterByBranchPrFallback`. Codex's post-merge challenge of #169 surfaced that `filterByPr` is still state-blind: a stale `merged`/`closed` manifest with matching stored pr_number can shadow a fresh `dispatched + null` run via the exact-pr selector path that bypasses the fallback predicate entirely. Filed as #170. The meta-rule extension: when applying state-machine-axis discipline to fix selector A, list every OTHER selector that feeds the same downstream consumer in the same code path (`filterByBranch`, `filterByPr`, `filterByBranchPrFallback`, `findManifestByRunId`) and audit each one's state-awareness explicitly. The "fix only the selector cited in the bug" pattern is the same compliance-theater tell as the original "fix only the state cited in the bug." Three iterations of the same pattern at three different selector layers should have made this visible by the second iteration; from #170 forward, the rubric for any resolver-selector fix MUST enumerate sibling selectors.

**Cross-iteration STOP signal (2026-04-13):** three consecutive codex post-merge challenges (`d738ed8` → `955cd6e` → `2e62895` → `c8ebb01`) each found a new-class bypass at one layer up. When the iteration count reaches 3+ on the same invariant, the rubric is structurally undersized — file the broader cross-cutting issue (or escalate to user) rather than continue auto-dispatching narrow fixes. The rubric author cannot enumerate every layer of a multi-layer invariant on the first try; the codex challenge IS the enumeration mechanism, and that's a process signal, not a rubric defect — but compounding PR count is a separate risk signal that requires user judgment on whether to continue, pause for an epic, or accept defensive gaps.

**Iteration-4 confirmation + scope-boundary trap (2026-04-13, post-merge challenge on `1dee17a`/PR #173/#170):** iteration 4 (#170) WAS the next ladder rung — the rubric explicitly applied the new selector-composition meta-rule, the file-level audit table got persisted, six factors graded out clean, codex-reviewer LGTM round 2. But the post-merge codex challenge surfaced the next class (standalone `--pr` selector at `relay-resolver.js:237` is still state-blind; reachable via `finalize-run --pr` and conditionally via `gate-check`). The rubric scoped factor 1 specifically to the "branch+PR call site" composition. That scope was the trap: when applying a new meta-rule on its first iteration, scope to the *invariant*, not to the *call site that surfaced the bug*. The factor should have been "every call site of `filterByPr` composes with a non-terminal set OR justifies inline why state-blindness is safe at that site." Specifically: when a single call site is fixed, the factor should explicitly require auditing every OTHER call site of the same selector in the SAME PR (factor 2 in #170's rubric did this for the four selectors as functions, but not for individual call sites of any one selector — `filterByPr` is called three times in `resolveManifestRecord` alone). The rubric extension: for any new-meta-rule's first iteration, expand sibling-enumeration to "every call site of the affected function in the file," not just "every sibling function." Memory rule update: the selector-composition axis enumeration meta-rule applies to **call sites** as well as to **selectors**. Iteration 5 — the user picked Options at this point; if iteration 5 fires, it must close the standalone `--pr` selector AND audit every call site of every resolver helper as a single sweep, not piecemeal.

**Fail-closed state-validation meta-rule (2026-04-14, from codex challenge on merged #174/PR #175):** selectors that EXCLUDE records by state must gate on a whitelist derived from the `STATES` enum, not on negation of a literal terminal blacklist, so tampered or missing `state` values fail-closed rather than fail-open.

**Why:** Rule 4 (state-machine-axis whitelist, 2026-04-12) established "whitelist not blacklist" on the state axis but was applied only to the state CITED IN THE BUG (merged/closed at #149, escalated at #165, any non-terminal + null pr at #168). Codex's post-merge challenge of `501eb8e` surfaced that #174's introduction of `BRANCH_ONLY_TERMINAL_STATES = {MERGED, CLOSED}` as a literal-match blacklist and inversion-as-exclusion (`!BRANCH_ONLY_TERMINAL_STATES.has(state)`) at two EXCLUSION sites — `filterOutTerminal()` and `filterByBranch({excludeTerminal: true})` — fails OPEN on tampered `state: "bogus"` or missing state: both sites ADMIT the stale record because it is not in the literal terminal set. Rule 4 was never applied to STATE PARSING itself. Exclusion-by-negation is structurally fail-open on anything not in the blacklist; exclusion-by-whitelist is structurally fail-closed on anything not in the whitelist. The asymmetry only reveals itself under manifest tampering or under a future state added to the enum but not to the exclusion set.

**How to apply:** every selector that filters records by state uses a whitelist predicate (`KNOWN_NON_TERMINAL_STATES.has(state)` or an `isNonTerminalState(state)` helper) rather than `!BRANCH_ONLY_TERMINAL_STATES.has(state)`. Define the whitelist by deriving from the enum: `KNOWN_NON_TERMINAL_STATES = new Set(Object.values(STATES).filter((s) => !BRANCH_ONLY_TERMINAL_STATES.has(s)))` — new states added to `STATES` are automatically admitted unless explicitly added to the blacklist, so the whitelist does not drift. DETECTION sites that positively ask "is this state terminal?" for classification (mixed-state detectors, UX branching on known terminal states, recovery-message state categorization) keep blacklist semantics by purpose and carry an inline comment naming meta-rule 7 AND the issues that introduced the site (e.g., `#170/#174/meta-rule 7`) so the next iteration does not misread the asymmetry as a state-blindness bug and flip it. Closed by #177/PR-pending.

**Timeout / contention-fallthrough policy split (2026-04-15, from codex post-merge challenge of merged #166/PR #184/`c450588`, probe 4):** lock or contention-fallthrough paths inside a multi-layer invariant must split their TIMEOUT POLICY by enforcement layer too — not just their happy-path logic. A unified "fail-safe on timeout" clause at the outer layer masks fail-safety breaks at the downstream gate.

**Why:** PR #184 (closes #166) applied meta-rule 1 (enforcement-layer split) correctly to the two HAPPY-PATH layers of gate-check's pr_number stamping — layer A (lock acquisition + write) and layer B (event-journal dedup) — but kept a UNIFIED timeout policy: on lock timeout, return the freshly read manifest and let gate-check continue. Codex's post-merge adversarial challenge probe 4 surfaced that the unified timeout policy was CORRECT for the audit-trail invariant (the next caller's layer B dedup handles late event emission so the audit stays clean) but WRONG for the merge-gate invariant: `evaluateReviewGate` at `review-gate.js:113` does not check `manifestData.git.pr_number`, so an unstamped manifest returned by the timeout path produced `{ status: "lgtm", readyToMerge: true }` — advancing the merge gate on a manifest that never received its first-resolution audit row. Same compliance-theater pattern as #138 rd5 / #155 rd1 (visible vs fail-closed): the unified outer-layer policy obscured a split-by-consumer inner-layer need. Filed and closed by #185.

**How to apply:** when authoring a rubric factor for a lock / contention-fallthrough path (O_EXCL + timeout, `Atomics.wait` + deadline, lockfile polling, similar), enumerate every DOWNSTREAM CONSUMER of the fallthrough output. For each consumer, ask: does this consumer require the same fail-policy the fallthrough itself chose? If any consumer requires a different policy (e.g., a merge gate requires fail-closed on a field the fallthrough makes no guarantee about, or an audit-trail consumer requires fail-safe where the fallthrough intended to fail-closed), split the rubric factor by downstream-consumer class and require separate targets for each. The fallthrough itself may be fail-safe OR fail-closed OR hybrid; the rubric must force the executor to choose per-consumer rather than adopt a single blanket policy. Generalizes beyond gate-check stamping: any lock-or-deadline-with-fallthrough that feeds multiple consumer layers applies. First application: #185's split of #166's unified fail-safe timeout into audit-trail (fail-safe, layer B) + merge-gate (fail-closed, layer A null-pr_number) + healthy-contention (unchanged) sub-policies.
